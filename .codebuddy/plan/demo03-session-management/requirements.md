# 需求文档

## 引言

本模块（demo03-session-management）是 Agent From Scratch 教学项目的第三个进阶 Demo，核心主题为**简单会话管理**。

### 背景

在现有的 demo02-gui-agent 中，Agent 的对话历史（`conversationHistory`）仅保存在内存中的一个 `ArrayList` 里，应用关闭后所有对话记录丢失，且只支持单一会话——用户无法创建新会话、切换历史会话或管理多个独立的对话上下文。

### 目标

demo03 在 demo02 的基础上，增加**简单会话管理**能力，让学习者理解以下核心概念：
- **会话（Session）** 的抽象与建模：每个会话拥有独立的对话历史和元数据
- **会话持久化**：将会话数据序列化到本地文件系统（JSON 格式），实现应用重启后恢复
- **多会话切换**：支持在 GUI 中创建、切换、删除多个独立会话
- **会话与 AgentLoop 的解耦**：会话管理层独立于 Agent 核心循环

### 技术约束

- 延续项目"无框架依赖、纯手搓"的风格，不引入数据库或第三方持久化框架
- 持久化采用 JSON 文件存储（利用已有的 Jackson 依赖）
- 继续使用 JavaFX 构建 GUI，在 demo02 的界面基础上扩展侧边栏
- 保持 Java 25+ / Maven 多模块结构

## 需求

### 需求 1：会话数据模型

**用户故事：** 作为一名学习者，我希望每个对话都有独立的会话实体，以便理解会话管理的数据建模方式。

#### 验收标准

1. WHEN 系统初始化 THEN 系统 SHALL 提供一个 `Session` 数据模型，包含以下字段：会话 ID（UUID）、会话标题、创建时间、最后更新时间、对话消息列表（`List<ChatMessage>`）
2. WHEN 创建新会话 THEN 系统 SHALL 自动生成唯一的会话 ID（UUID 格式）并设置创建时间
3. WHEN 会话中产生第一条用户消息 THEN 系统 SHALL 自动将该消息内容的前 20 个字符作为会话标题（支持后续手动修改）
4. WHEN 会话中有新消息追加 THEN 系统 SHALL 自动更新该会话的"最后更新时间"字段

### 需求 2：会话持久化存储

**用户故事：** 作为一名用户，我希望关闭应用后重新打开时能恢复之前的对话记录，以便继续之前的工作。

#### 验收标准

1. WHEN 会话数据发生变更（新增消息、创建/删除会话） THEN 系统 SHALL 将变更后的会话数据以 JSON 格式持久化到本地文件系统
2. WHEN 应用启动 THEN 系统 SHALL 从本地存储目录加载所有已保存的会话数据，并恢复到内存中
3. IF 本地存储目录不存在 THEN 系统 SHALL 自动创建该目录
4. IF 本地存储文件损坏或格式错误 THEN 系统 SHALL 记录警告日志并跳过该文件，不影响其他会话的加载
5. WHEN 持久化存储时 THEN 系统 SHALL 采用"每个会话一个 JSON 文件"的方式，文件名为 `{sessionId}.json`，存储在用户主目录下的 `.agent-from-scratch/sessions/` 目录中

### 需求 3：多会话管理

**用户故事：** 作为一名用户，我希望能够创建多个独立的对话会话并在它们之间自由切换，以便针对不同任务使用不同的对话上下文。

#### 验收标准

1. WHEN 用户点击"新建会话"按钮 THEN 系统 SHALL 创建一个新的空白会话，并自动切换到该会话
2. WHEN 用户选择一个历史会话 THEN 系统 SHALL 切换到该会话，并在聊天区域展示该会话的完整对话历史
3. WHEN 用户切换会话 THEN 系统 SHALL 将当前 AgentLoop 的对话上下文替换为目标会话的消息列表
4. WHEN 用户删除一个会话 THEN 系统 SHALL 从内存和本地存储中移除该会话数据
5. IF 用户删除的是当前活跃会话 THEN 系统 SHALL 自动切换到最近的其他会话；IF 没有其他会话 THEN 系统 SHALL 自动创建一个新的空白会话
6. WHEN 应用启动且存在已保存的会话 THEN 系统 SHALL 自动加载并选中最近更新的会话

### 需求 4：GUI 侧边栏（会话列表）

**用户故事：** 作为一名用户，我希望在聊天界面左侧看到一个会话列表侧边栏，以便直观地管理和切换多个会话。

#### 验收标准

1. WHEN 应用启动 THEN 系统 SHALL 在聊天窗口左侧显示一个会话列表侧边栏，展示所有会话的标题和最后更新时间
2. WHEN 会话列表中有多个会话 THEN 系统 SHALL 按最后更新时间倒序排列（最新的在最上面）
3. WHEN 用户点击侧边栏中的某个会话项 THEN 系统 SHALL 高亮选中该会话并切换聊天区域内容
4. WHEN 侧边栏顶部 THEN 系统 SHALL 显示一个"+ 新建会话"按钮
5. WHEN 用户右键点击会话项 THEN 系统 SHALL 显示上下文菜单，包含"删除会话"选项
6. WHEN 当前活跃会话 THEN 系统 SHALL 在侧边栏中以视觉高亮方式标识

### 需求 5：会话与 AgentLoop 的集成

**用户故事：** 作为一名学习者，我希望理解会话管理层如何与 Agent 核心循环解耦集成，以便学习分层架构设计。

#### 验收标准

1. WHEN 切换到一个已有对话历史的会话 THEN 系统 SHALL 用该会话的消息列表重建 AgentLoop 的对话上下文，使 Agent 能够基于历史上下文继续对话
2. WHEN 切换到一个新的空白会话 THEN 系统 SHALL 重置 AgentLoop 的对话上下文（仅保留 system prompt）
3. WHEN Agent 产生新的回复（包括工具调用和最终回复） THEN 系统 SHALL 将新消息同步追加到当前活跃会话的消息列表中
4. WHEN 会话管理层与 AgentLoop 交互 THEN 系统 SHALL 通过 SessionManager 服务层进行，不直接操作 AgentLoop 内部状态

### 需求 6：教学代码质量

**用户故事：** 作为一名学习者，我希望代码结构清晰、注释充分，以便通过阅读代码学习会话管理的实现原理。

#### 验收标准

1. WHEN 编写代码 THEN 系统 SHALL 遵循 demo02 的代码风格：类和方法均有中文 Javadoc 注释，关键逻辑有行内注释
2. WHEN 组织代码结构 THEN 系统 SHALL 新增 `session` 包，包含 `Session`（数据模型）、`SessionManager`（会话管理服务）、`SessionStore`（持久化存储）等类
3. WHEN 设计模块结构 THEN 系统 SHALL 保持与 demo02 一致的分层架构风格（model / core / agent / ui / session）
